  PHASE 1 — INTENT DISCOVERY

  The Real Goal

  This system is Ibrahim's automated client acquisition engine for his freelance 3D/CGI/motion design business operating as "Viral X" studio out of Dubai. The objective is not data collection — it is revenue
  pipeline generation. Every lead is a potential client who needs 3D visualization, motion design, CGI production, or DOOH content right now.

  What Type of Leads It Targets

  The system hunts for demand signals — evidence that a company currently needs 3D/creative services but hasn't yet filled that need. Four signal types:

  1. Active hiring signals: Companies posting 3D Artist / Motion Designer / CGI roles on LinkedIn and ArtStation. The thesis: if they're hiring full-time, they need help today and won't fill that seat for 2-4
  months. A freelancer can fill the gap immediately.
  2. Funding signals: MENA startups that just raised Seed/Series A. Post-funding startups scramble to build marketing collateral. They have cash but no creative team.
  3. Project signals: Companies posting explicit 3D/animation/CGI project briefs on Upwork. Direct evidence of budget and intent.
  4. Expansion signals (designed but not implemented): New office openings, key person departures, competitor mimicry — all theorized in the Strategy Framework but never automated.

  Intended Outcome of the Excel Output

  The Excel file ibrahim_guerrilla_targeting_v2.xlsx is intended to be a live intelligence database that feeds a manual outreach workflow:

  1. Strategy Framework sheet defines what to look for
  2. Query Library sheet defines how to look
  3. Regional Discovery sheet defines where to look
  4. Company Tracker sheet is the master CRM — the canonical list of all discovered leads
  5. Source-specific sheets (ArtStation, LinkedIn, Wamda, Upwork) are the raw intake funnels

  The human workflow is: run the suite → sort by priority → research A+ leads → craft personalized outreach → pitch via LinkedIn/email.

  Assumptions the Current Architecture Makes

  1. CSS selectors are stable — assumes platform DOM structures don't change (they do, frequently)
  2. Gemini CLI always returns valid JSON — a single malformed response silently falls back to raw data with no flag
  3. Company name is a reliable dedup key — "ACME Inc" vs "ACME" vs "Acme Inc." will all be treated as different entities
  4. Excel is a viable database — concurrent writes, schema enforcement, query capability are all absent
  5. Sequential execution is acceptable — 7 regions × 15 queries × 60-second timeouts = potentially 1.75+ hours for LinkedIn alone
  6. All scraped data is equally valuable — no confidence scoring, no freshness decay, no signal strength measurement
  7. LLM enrichment is fire-and-forget — no validation of LLM output quality, no retry on low-confidence results

  Artificial vs Necessary Constraints

  Artificial (can be removed):
  - Hardcoded region list in strategy_loader.py despite the Excel sheet containing dynamic regions
  - Arbitrary query caps ([:15] LinkedIn, [:5] Upwork) with no intelligence behind the limits
  - Single-threaded sequential scraping
  - 20-second Gemini timeout with no retry
  - Excel as sole data store
  - No scheduling or incremental run capability

  Necessary (must work around):
  - LinkedIn anti-bot protections (StealthyFetcher is the right tool)
  - Cloudflare on Upwork (solve_cloudflare=True is correct)
  - Gemini CLI as the local LLM interface (no API, subprocess-only)
  - Rate limiting on target platforms
  - Excel as the output format (Ibrahim's manual review happens there)

  Specific Structural Inefficiencies

  1. Strategy Loader is a Facade
  strategy_loader.py claims to "parse the Master Excel sheet for regions and Boolean queries" but hardcodes all 7 regions and only extracts queries containing "linkedin" + "job" or "upwork" in the platform
  column. The Query Library has 100+ entries spanning 12 platforms — 95% are ignored.

  2. LLM Enricher is a Single-Shot Gamble
  parse_with_gemini() fires a single subprocess call with a 20-second timeout. If the LLM returns malformed JSON, the raw dict is silently used. There is:
  - No schema validation of the response
  - No confidence threshold
  - No retry on failure
  - No hallucination detection
  - No cross-field consistency check
  - No batching (each lead = 1 subprocess call = 1 cold start)

  For LinkedIn's 16,500 leads, this means potentially 16,500 individual gemini.cmd invocations — each spawning a new process.

  3. Deduplication is String-Match Theatre
  The dedup logic in utils.py compares str(val).lower().strip() — pure string equality. No normalization of:
  - Company suffixes (Inc, LLC, Ltd, GmbH)
  - Unicode variants
  - Abbreviations (Int'l vs International)
  - Domain-based identity (two entries with same website are still treated as different if names differ)

  4. Excel as Database is a Time Bomb
  - No schema enforcement — any column can contain any value
  - No referential integrity between source sheets and master tracker
  - Concurrent writes (if multiple scrapers ran in parallel) would corrupt the file
  - openpyxl loads the entire file into memory for every save operation
  - No indexing — dedup requires full column scan every time
  - File grows unboundedly with no archival strategy

  5. Priority Scoring is Non-Functional
  LinkedIn scraper: every lead gets the LLM to determine priority. But the leads_linkedin.json shows all 16,500 entries marked "A". The LLM is not discriminating because:
  - The prompt provides no calibration examples
  - There's no rubric for what differentiates A+ from A from B
  - There's no budget/urgency signal in the data being sent to the LLM
  - The fallback priority is "A" anyway

  6. Upwork is Structurally Orphaned
  Upwork leads use title as key (not company), are excluded from the master Company Tracker, and have a completely different schema. They exist in a parallel universe within the same system.

  7. No Contact Intelligence
  The system finds companies but extracts zero contact information. No:
  - Decision maker names
  - Email addresses
  - LinkedIn profile URLs
  - Phone numbers
  - Company website extraction (only ArtStation attempts this)

  The outreach workflow requires manually researching every lead — the system generates awareness but not actionable intelligence.

  Concurrency Model

  Async task queue with per-source semaphores:

  - Global orchestrator dispatches scrape tasks to an asyncio-compatible queue
  - Each source has its own semaphore limiting concurrent requests (defined in registry)
  - Within a source, requests are serialized with jittered delays
  - Between sources, requests are fully parallel
  - Worker pool size: sum(max_concurrency for all active sources)

  This means LinkedIn (2 concurrent) + Upwork (1) + ArtStation (3) + Wamda (2) = 8 concurrent sessions max.

  Anti-Detection Strategy

  Layer 1 — Scrapling-Native:
  - StealthyFetcher with real browser profiles (already handles fingerprint randomization)
  - solve_cloudflare=True where needed

  Layer 2 — Behavioral:
  - Jittered delays between requests: random.uniform(*base_delay)
  - Session reuse within a source (don't create new browser per request)
  - Limit pages per query (max 3-5 pages before moving to next query)
  - Randomize query execution order per run

  Layer 3 — Circuit Breaker:
  - Track consecutive failures per source
  - If failure_threshold exceeded → pause source for escalating cooldown (1min, 5min, 15min)
  - Log circuit break events for human review
  - Never hammer a blocked source

  Retry & Fallback Strategy

  FETCH ATTEMPT
    ├── Success (200) → Extract → Continue
    ├── Soft Block (403, captcha page) → Retry with backoff (max 3)
    │     ├── Still blocked → Circuit break this source
    │     └── Recovered → Continue with extended delays
    ├── Rate Limited (429) → Exponential backoff (30s, 60s, 120s)
    ├── Timeout → Retry once with 2x timeout
    ├── Network Error → Retry once after 10s
    ├── Selector Miss (200 but 0 results) →
    │     ├── Save debug HTML with timestamp
    │     ├── Flag as SELECTOR_DRIFT
    │     └── Continue (don't retry — the page loaded, selectors are wrong)
    └── Unknown Error → Log full traceback, skip, continue

  Structured DOM Extraction Rules

  Each source module defines an ExtractionContract:

  @dataclass
  class ExtractionField:
      name: str                     # Field name in output
      selectors: list[str]          # CSS selector chain (fallback order)
      attribute: str = "text"       # "text", "href", "datetime", etc.
      required: bool = False        # If True, skip entire record on miss
      transform: Callable = None    # Post-extraction transform function

  @dataclass
  class ExtractionContract:
      source: str
      container_selector: str       # The repeating element (job card, article)
      container_fallbacks: list[str]
      fields: list[ExtractionField]
      min_expected_results: int = 1 # Alert if fewer results than this

  This replaces the current approach of ad-hoc job.css("h3.job-tile-title a").first scattered throughout each scraper. The contract is declarative, testable, and version-controlled.

  Pre-LLM Normalization Layer

  Before any data touches the LLM, apply deterministic transformations:

  1. Whitespace normalization — collapse multi-spaces, strip, remove zero-width chars
  2. HTML entity decoding — &amp; → &, etc.
  3. Location standardization — known mapping table: UAE → United Arab Emirates, KW → Kuwait, etc.
  4. Date normalization — parse "3 days ago", "2024-01-15", "Jan 15", "15/01/2024" all to ISO 8601
  5. URL cleaning — strip tracking params (?utm_*, ?trk=, ?refId=)
  6. Budget parsing — "$1,000 - $5,000" → {"min": 1000, "max": 5000, "currency": "USD"}
  7. Empty field standardization — all empty/null/whitespace → explicit None

  This layer is pure Python. Zero LLM calls. Reduces LLM workload and error surface.

  Data Validation Before Enrichment

  Post-extraction, pre-LLM gate:

  @dataclass
  class RawLead:

      company: str | None
      title: str | None
      location_raw: str | None
      url: str | None
      posted_raw: str | None
      description: str | None
      budget_raw: str | None
      metadata: dict

      def is_valid_for_enrichment(self) -> tuple[bool, str]:
          """Returns (valid, reason) — only valid leads enter LLM pipeline."""
          if not self.company and not self.title:
              return False, "NO_IDENTIFIABLE_ENTITY"
          if self.url and not self.url.startswith("http"):
              return False, "INVALID_URL"
          return True, "OK"

  Leads that fail validation are logged to a rejected_leads.jsonl file with reason codes. They never consume LLM tokens.

  Pass 1: Extraction

  Goal: Clean raw scraped data into canonical form. LLM is used ONLY when deterministic parsing fails.

  # Deterministic first, LLM fallback
  def extract_location(raw_location: str) -> dict:
      """Try regex/lookup first, LLM only if needed."""
      # Attempt 1: Direct lookup
      for country, patterns in COUNTRY_PATTERNS.items():
          if any(p in raw_location.lower() for p in patterns):
              return {"country": country, "city": extract_city(raw_location, country)}

      # Attempt 2: LLM extraction (only for truly ambiguous strings)
      return llm_extract_location(raw_location)

  Batching: Instead of 1 subprocess call per lead, batch up to 10 leads per LLM invocation:

  EXTRACTION_PROMPT = """
  You are a data extraction engine. Parse each entry below into the exact JSON schema.

  Entries:
  {batch_json}

  Return a JSON array of objects, one per entry. Schema per object:
  {{
    "id": <entry index>,

  Pass 2: Enrichment

  Goal: Add intelligence that requires world knowledge. This is where the LLM earns its keep.

  ENRICHMENT_PROMPT = """

  For each company below, provide enrichment based on your knowledge:

  {batch_json}

  {{

  - demand_urgency: "immediate" = active job post or recent funding. "short_term" = likely to need soon. "exploratory" = speculative.
  - If you don't recognize the company, base assessment purely on the job title and description provided.
  - Do NOT hallucinate company details you don't know. Use the data provided.
  """

  Pass 3: Validation

  Goal: Catch LLM errors before they enter the database. This pass is primarily deterministic with one LLM self-consistency check.

  Deterministic checks:
  def validate_enriched_lead(lead: dict) -> tuple[bool, list[str]]:
      issues = []

      # Cross-field: country in URL should match declared country
      if lead.get("url") and lead.get("country"):
          tld = extract_tld(lead["url"])
          if tld == ".ae" and lead["country"] != "United Arab Emirates":
              issues.append("COUNTRY_URL_MISMATCH")

      # Range check: scores must be 0-1
      if not (0 <= lead.get("service_match_score", 0) <= 1):
          issues.append("SCORE_OUT_OF_RANGE")

      # Hallucination flag: if LLM returned very specific details for an unknown company
      if lead.get("estimated_size") == "enterprise" and lead.get("source") == "Upwork":
          issues.append("UNLIKELY_SIZE_FOR_SOURCE")

      # Self-consistency: company name should not be a sentence
      if len(lead.get("company_clean", "").split()) > 6:
          issues.append("COMPANY_NAME_TOO_LONG")

      return len(issues) == 0, issues

  Confidence scoring:
  def compute_confidence(lead: dict, validation_issues: list) -> float:
      """0.0 to 1.0 confidence that this lead is high-quality and accurate."""
      score = 1.0

      # Penalize for each validation issue
      score -= len(validation_issues) * 0.15

      # Penalize for "Unknown" fields
      unknown_count = sum(1 for v in lead.values() if v in ("Unknown", None, ""))
      score -= unknown_count * 0.05

      # Boost for explicit signals
      if lead.get("demand_urgency") == "immediate":
          score += 0.1
      if lead.get("url") and lead["url"].startswith("http"):
          score += 0.05

      return max(0.0, min(1.0, score))

  Pass 4: Scoring

  Goal: Deterministic priority assignment based on a calibrated rubric. NO LLM in this pass — the LLM already provided the inputs in Pass 2.

  SCORING_RUBRIC = {
      # Signal type weights
      "active_hiring": 30,          # Company has an open role for 3D/CGI
      "recent_funding": 25,         # Raised money in last 6 months
      "direct_project": 35,         # Posted a specific 3D project (Upwork)
      "adjacent_hiring": 15,        # Hiring for related creative role

      # Urgency multipliers
      "immediate": 1.5,
      "short_term": 1.0,
      "exploratory": 0.5,

      # Fit multipliers
      "service_match_threshold_a_plus": 0.8,
      "service_match_threshold_a": 0.5,
      "service_match_threshold_b": 0.3,

      # Geography bonus (MENA = home market)
      "mena_bonus": 10,
      "europe_bonus": 5,
  }

  def score_lead(lead: dict) -> dict:
      base = 0

      # Signal type
      if "hiring" in lead.get("demand_signal", "").lower():
          base += SCORING_RUBRIC["active_hiring"]
      if "fund" in lead.get("demand_signal", "").lower():
          base += SCORING_RUBRIC["recent_funding"]
      if lead.get("source") == "Upwork":
          base += SCORING_RUBRIC["direct_project"]

      # Urgency
      urgency = lead.get("demand_urgency", "exploratory")
      base *= SCORING_RUBRIC.get(urgency, 1.0)

      # Fit
      fit = lead.get("service_match_score", 0.5)
      base *= fit

      # Geography
      country = lead.get("country", "")
      if country in MENA_COUNTRIES:
          base += SCORING_RUBRIC["mena_bonus"]
      elif country in EUROPE_COUNTRIES:
          base += SCORING_RUBRIC["europe_bonus"]

      # Priority assignment
      if base >= 35:
          priority = "A+"
      elif base >= 20:
          priority = "A"
      elif base >= 10:
          priority = "B+"
      else:
          priority = "B"

      return {
          "priority": priority,
          "score_raw": round(base, 2),
          "score_breakdown": {...}  # Full explanation
      }

  JSON Contract Enforcement

  Every LLM call is wrapped in a strict contract validator:

  def call_gemini_with_contract(
      prompt: str,
      expected_schema: dict,      # JSON Schema definition
      max_retries: int = 3,
      timeout: int = 30
  ) -> tuple[dict | list, float]:  # Returns (parsed_data, confidence)

      for attempt in range(max_retries):
          raw_output = invoke_gemini_cli(prompt, timeout)
          parsed = attempt_json_parse(raw_output)

          if parsed is None:
              # Retry with explicit "RETURN ONLY JSON" reinforcement
              continue

          # Validate against schema
          valid, errors = validate_against_schema(parsed, expected_schema)

          if valid:
              return parsed, 1.0 - (attempt * 0.1)

          if attempt < max_retries - 1:
              # Include validation errors in retry prompt
              prompt += f"\n\nYour previous response had errors: {errors}. Fix them."

      # All retries exhausted
      return None, 0.0

  Entity Resolution

  The dedup problem requires a multi-signal approach:

  def resolve_entity(new_lead: dict, existing_leads: list[dict]) -> str | None:
      """Returns ID of matching existing lead, or None if new."""

      candidates = []

      for existing in existing_leads:
          score = 0.0

          # Signal 1: Exact name match (normalized)
          if normalize_company(new_lead["company"]) == normalize_company(existing["company"]):
              score += 0.5

          # Signal 2: Fuzzy name match (Levenshtein ratio > 0.85)
          elif fuzz_ratio(new_lead["company"], existing["company"]) > 0.85:
              score += 0.35

          # Signal 3: Domain match
          if new_lead.get("website") and existing.get("website"):
              if extract_domain(new_lead["website"]) == extract_domain(existing["website"]):
                  score += 0.4

          # Signal 4: URL overlap
          if new_lead.get("url") == existing.get("url"):
              score += 0.8  # Same exact listing

          # Signal 5: Same location + similar name
          if (new_lead.get("city") == existing.get("city") and
              fuzz_ratio(new_lead["company"], existing["company"]) > 0.7):
              score += 0.2

          if score >= 0.5:
              candidates.append((existing["id"], score))

      if candidates:
          return max(candidates, key=lambda x: x[1])[0]
      return None

  Hallucination Mitigation

  1. Never ask the LLM for facts it can't know — don't ask "what is the company revenue?" for obscure startups
  2. Extraction prompts explicitly say "do NOT invent" — reinforced in every prompt
  3. Cross-field validation catches impossible combinations — a 2-person startup marked "enterprise"
  4. Confidence cascading — low confidence in Pass 1 → skip Pass 2 → mark lead for manual review
  5. Reasoning traces — every LLM enrichment includes a "reasoning" field that is stored for audit
  6. Batch processing with indexing — every input/output pair is tracked by index, preventing misalignment

  Retry Logic Based on Confidence Thresholds

  Confidence >= 0.8  → Accept lead as-is
  Confidence 0.5-0.8 → Retry enrichment with more specific prompt (1 retry)
  Confidence 0.3-0.5 → Accept but flag as LOW_CONFIDENCE in data
  Confidence < 0.3   → Reject from automated pipeline, route to manual review queue

  PHASE 4 — SYSTEM PRINCIPLES

  1. Data Contracts Between Layers

  SCRAPER OUTPUT → RawLead (dataclass)
      ├── All fields are str | None
      ├── No LLM-derived fields
      ├── Must include source + URL
      └── Validated by is_valid_for_enrichment()

  NORMALIZER OUTPUT → NormalizedLead (dataclass)
      ├── Dates are ISO 8601
      ├── Locations are standardized
      ├── URLs are cleaned
      └── Empty fields are explicit None

  LLM EXTRACTION OUTPUT → ExtractedLead (dataclass)
      ├── All fields from NormalizedLead
      ├── + company_clean, country, city
      ├── Validated against JSON Schema
      └── Includes extraction_confidence: float

  LLM ENRICHMENT OUTPUT → EnrichedLead (dataclass)
      ├── All fields from ExtractedLead
      ├── + industry, estimated_size, service_match_score, demand_urgency
      ├── + reasoning: str
      ├── Validated against JSON Schema
      └── Includes enrichment_confidence: float

  SCORER OUTPUT → ScoredLead (dataclass)
      ├── All fields from EnrichedLead
      ├── + priority, score_raw, score_breakdown
      ├── Deterministic — no LLM involvement
      └── Includes overall_confidence: float

  FINAL OUTPUT → Lead (dataclass)
      ├── All fields from ScoredLead
      ├── + lead_id, status, first_seen, last_updated, run_id
      ├── + validation_issues: list[str]
      └── Ready for storage

  Rule: Each layer accepts ONLY the output of the previous layer's dataclass. No shortcuts. No passing raw dicts between stages.

  2. Deterministic vs Probabilistic Boundaries

  ┌────────────────┬───────────────┬───────────────────────────────────────┐
  │     Layer      │    Nature     │               Guarantee               │
  ├────────────────┼───────────────┼───────────────────────────────────────┤

  My proposed fetcher_pool.py that manages session lifecycle is redundant. Scrapling's async sessions already do this.

  2. I proposed custom retry/backoff — Scrapling has built-in retries

  Every fetcher accepts retries (default: 3) and retry_delay (default: 1s). The HTTP fetchers also auto-detect proxy errors and retry with the next proxy via ProxyRotator. My custom retry strategy still has
  value for circuit breaking (which Scrapling doesn't do), but the low-level retry logic should delegate to Scrapling, not reimplement it.

  3. I proposed custom proxy rotation — Scrapling has ProxyRotator

  4. I completely missed Adaptive Element Tracking — this solves selector drift

  Scrapling has a built-in system where you can save element identities and relocate them later even if the DOM structure changes. This directly addresses the SELECTOR_DRIFT failure class I flagged:

  import asyncio
  import random

  async def scrape_source(source_name: str, queries: list[dict], config: dict):
      """Scrapes one source using Scrapling's native async session with tab pooling."""
      results = []
      consecutive_failures = 0

      async with await create_session(config) as session:
          for query in queries:
              # Circuit breaker check
              if consecutive_failures >= config["circuit_breaker_threshold"]:
                  logger.critical(f"CIRCUIT_BREAK: {source_name} after {consecutive_failures} failures")
                  break

              # Behavioral delay (the only layer WE manage)
              delay = random.uniform(*config["delay_range"])
              await asyncio.sleep(delay)

              try:
                  for page_num in range(1, config["max_pages_per_query"] + 1):
                      url = build_url(source_name, query, page_num)

                      # Scrapling handles: anti-detection, retries, proxy rotation, tab management
                      response = await session.fetch(url)

                      if response.status != 200:
                          consecutive_failures += 1
                          logger.warn(f"HTTP {response.status} on {url}")
                          continue

                      leads = extract_leads(response, source_name, config)

                      if not leads and page_num == 1:
                          # First page returned 0 results — possible selector drift
                          save_debug_html(response, source_name, query)
                          logger.warn(f"SELECTOR_DRIFT_SUSPECT: {source_name}, query={query}")
                          consecutive_failures += 1
                          break

                      results.extend(leads)
                      consecutive_failures = 0  # Reset on success

                      if len(leads) < expected_per_page(source_name):
                          break  # No more pages

              except Exception as e:
                  consecutive_failures += 1
                  logger.error(f"FETCH_ERROR: {source_name}: {e}")

      return results

  def extract_leads(response, source_name: str, config: dict) -> list[RawLead]:
      """Uses Scrapling's rich selector API — not just .css()."""
      contract = EXTRACTION_CONTRACTS[source_name]

      # Primary: CSS with adaptive tracking (Scrapling relocates if DOM changed)
      containers = response.css(
          contract.container_selector,
          identifier=f"{source_name}_container",
          adaptive=config.get("adaptive_tracking", False),
          auto_save=True
      )

      # Fallback: try alternate selectors
      if not containers:
          for fallback in contract.container_fallbacks:
              containers = response.css(fallback)
              if containers:
                  logger.info(f"Used fallback selector for {source_name}: {fallback}")
                  break

      # Last resort: find by text pattern (Scrapling-native)
      if not containers and contract.text_pattern:
          containers = response.find_by_regex(
              contract.text_pattern,
              first_match=False
          )

      leads = []
      for container in containers:
          lead_data = {}

          for field in contract.fields:
              value = None

              for selector in field.selectors:
                  # Use CSS within the container
                  elements = container.css(selector)
                  if elements:
                      if field.attribute == "text":
                          value = elements[0].text
                      elif field.attribute == "href":
                          value = elements[0].attrib.get("href")
                          if value:
                              value = response.urljoin(str(value))  # Scrapling's urljoin
                      elif field.attribute == "all_text":
                          value = elements[0].get_all_text(
                              separator=" ", strip=True,
                              ignore_tags=("script", "style")
                          )
                      break

              if value is None and field.required:
                  break  # Skip this record — required field missing

              lead_data[field.name] = str(value).strip() if value else None
          else:
              # All required fields present
              leads.append(RawLead(source=source_name, **lead_data))

      return leads
